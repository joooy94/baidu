## 1. 系统架构总览

### 1.1 数据流

```
CSV原始数据 (T01~T60格式)
        ↓
  config.py              点位映射 (tagtable + tagindex → 传感器/设备)
        ↓
  data_loader.py         CSV读取 → T01~T60展开为分钟级时间序列 → 异常值标记
        ↓
  ┌─────────────────────────────────────────────────┐
  │  sensor_diagnosis.py    传感器级诊断(缺失/漂移/异跳)  │
  │  health_assessment.py   投矾泵健康度评分              │
  │  equipment_diagnosis.py 设备级诊断(泵/管道/阀门/池/通讯) │
  └─────────────────────────────────────────────────┘
        ↓
  visualization.py       生成图表 (雷达图/柱状图/趋势图/饼图)
```

## 2. 数据加载模块

**文件**: `data_loader.py`

### 2.1 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `tagtable` | `str` | 数据表名，如 `"ZHJY"`, `"ShuiZhi"` |
| `tagindexes` | `list[int]` | 要加载的点位编号列表 |
| `date_range` | `tuple[str, str]` | 日期范围，格式 `"YYYY-MM-DD"` |

**原始数据格式**（CSV）:

```
Bid, TagIndex, DateDay, DateHour, T01, T02, ..., T60
```

每行 = 1个传感器点位 × 1小时 × 60个分钟级采样值。

### 2.2 处理流程

1. **查找CSV文件**: 根据 `tagtable` 从 `config.CSV_FILES` 获取对应CSV文件路径列表
2. **读取CSV**: `pd.read_csv()` 加载，按 `tagindexes` 和 `date_range` 过滤
3. **展开时间序列**: 每行 `T01~T60` 展开为60条记录：
   - `T01` → 该小时第0分钟，`T02` → 第1分钟，...，`T60` → 第59分钟
   - 时间戳 = `DateDay` + `DateHour` + `minute_offset`
4. **异常值标记**: 将 `-4.9991`、`-4.999`、`-9999` 替换为 `NaN`，设置 `is_valid=False`

### 2.3 输出

**DataFrame**，列为：

| 列名 | 类型 | 说明 |
|---|---|---|
| `timestamp` | `datetime` | 分钟级时间戳 |
| `value` | `float` | 数值，异常值已替换为 `NaN` |
| `is_valid` | `bool` | 是否有效数据 |

### 2.4 上层接口

| 函数 | 返回值 | 说明 |
|---|---|---|
| `load_csv_data(tagtable, tagindexes, date_range)` | `DataFrame[timestamp, tagindex, value]` | 底层加载+展开，未标记异常值 |
| `load_sensor_data(sensor_type, sensor_name, date_range)` | `dict[str, DataFrame]` | 按传感器名称分组，已标记异常值 |
| `load_pump_data(pump_id, date_range)` | `dict[str, DataFrame]` | 按 `flow/error/auto/remote` 分组，已标记异常值 |
| `mark_invalid_values(df)` | `DataFrame` | 将异常标记值替换为NaN，添加 `is_valid` 列 |

## 3. 传感器诊断模块

**文件**: `sensor_diagnosis.py`

### 3.1 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `sensor_type` | `str` | 传感器类型：`"出水浊度仪"` / `"进水浊度仪"` / `"储液池液位计"` / `"电磁流量计"` |
| `sensor_name` | `str \| None` | 传感器名称，`None` 则诊断该类型全部传感器 |
| `date_range` | `tuple[str, str]` | 日期范围 |

数据通过 `data_loader.load_sensor_data()` 自动加载。

### 3.2 输出

**`list[DiagnosisResult]`**，每条结果包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `sensor_type` | `str` | 传感器类型 |
| `sensor_name` | `str` | 传感器名称 |
| `anomaly_type` | `str` | 异常类型：`data_absence` / `data_drift` / `data_jump` |
| `alarm_level` | `int` | 报警等级：1(一级/最高) / 2(二级) / 3(三级/最低) |
| `alarm_level_name` | `str` | 等级名称：`"一级"` / `"二级"` / `"三级"` |
| `start_time` | `datetime` | 异常开始时间 |
| `end_time` | `datetime` | 异常结束时间 |
| `duration_min` | `int` | 持续时间（分钟） |
| `detail` | `str` | 详细描述 |
| `threshold` | `float` | 触发阈值 |
| `actual_value` | `float` | 实际检测值 |

结果按 `alarm_level` 升序排列（一级优先）。

### 3.3 三种异常检测算法

#### 3.3.1 数据缺失检测 `detect_data_absence()`

**算法**：
1. 标记所有 `is_valid=False` 的数据点
2. 按连续性分组（相邻缺失点为一组）
3. 计算每组持续时间（=缺失点数量，因为采样间隔1分钟）
4. 从高等级到低等级匹配阈值

**阈值**（对应设计文档表11）：

| 报警等级 | 连续缺失时长 | 与设计文档一致性 |
|---|---|---|
| 三级 | ≥10分钟 | ✅ 完全一致 |
| 二级 | ≥30分钟 | ✅ 完全一致 |
| 一级 | ≥60分钟 | ✅ 完全一致 |

**匹配逻辑**：遍历 `[1, 2, 3]`（从高到低），第一个满足的等级即为报警等级。例如连续35分钟缺失 → 先检查≥60min(不满足) → ≥30min(满足) → 二级报警。

#### 3.3.2 数据漂移检测 `detect_data_drift()`

**算法**：
1. 计算前30分钟滑动窗口均值 `hist_mean`
2. 计算当前值与 `hist_mean` 的偏差（百分比或绝对值）
3. 超过阈值的点按连续性分组
4. 每组持续时间 ≥ `duration_min` 则报警

**阈值**（对应设计文档表11）：

| 传感器类型 | 报警等级 | 偏差阈值 | 持续时间 | 与设计文档一致性 |
|---|---|---|---|---|
| 电磁流量计 | 三级 | ±20% | ≥10min | ✅ 完全一致 |
| 电磁流量计 | 二级 | ±40% | ≥5min | ✅ 完全一致 |
| 电磁流量计 | 一级 | ±50% | ≥3min | ✅ 完全一致 |
| 出水浊度仪 | 三级 | ±15% | ≥10min | ✅ 完全一致 |
| 出水浊度仪 | 二级 | ±40% | ≥5min | ✅ 完全一致 |
| 出水浊度仪 | 一级 | ±50% | ≥3min | ✅ 完全一致 |
| 进水浊度仪 | 三级 | ±15% | ≥10min | ✅ 完全一致 |
| 进水浊度仪 | 二级 | ±40% | ≥5min | ✅ 完全一致 |
| 进水浊度仪 | 一级 | ±50% | ≥3min | ✅ 完全一致 |
| 储液池液位计 | 二级 | 绝对偏差>10cm | ≥1min | ✅ 完全一致 |

**偏差计算公式**：
- 百分比模式：`deviation_pct = |value - hist_mean| / hist_mean`
- 绝对值模式（液位计）：`deviation = |value - hist_mean|`（cm）

#### 3.3.3 数据异跳检测 `detect_data_jump()`

**算法**：计算相邻采样点间的变化率（`|value[t] - value[t-1]|`），与阈值对比。

**阈值**（对应设计文档表11）：

| 传感器类型 | 检测模式 | 阈值 | 与设计文档一致性 |
|---|---|---|---|
| 出水浊度仪 | 固定阈值 | >50 NTU/min，连续≥3个采样周期 | ✅ 完全一致 |
| 进水浊度仪 | 正常波动倍数 | >正常波动的3倍 | ✅ 完全一致 |
| 电磁流量计 | 正常波动倍数 | >正常波动的3倍 | ✅ 完全一致 |
| 储液池液位计 | 10分钟窗口变化 | >30cm/10min | ✅ 完全一致 |

**正常波动倍数**的计算：`threshold = std(rate) * normal_multiple`，其中 `rate = |diff(value)|`。

## 4. 健康度评估模块

**文件**: `health_assessment.py`

### 4.1 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `pump_id` | `str` | 泵编号：`"P1"` ~ `"P5"` |
| `date_range` | `tuple[str, str]` | 日期范围 |

数据通过 `data_loader.load_pump_data()` 加载，获取 `flow`、`error`、`auto`、`remote` 四组数据。

### 4.2 输出

**`PumpHealthResult`**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `pump_id` | `str` | 泵编号 |
| `overall_score` | `float` | 综合健康度（0~100） |
| `grade` | `str` | 等级：优秀/良好/轻微异常/异常 |
| `indicators` | `list[HealthIndicator]` | 四维指标明细 |
| `timestamp_range` | `tuple` | 评估数据时间范围 |
| `warnings` | `list[str]` | 告警信息 |

**`HealthIndicator`**（单个指标）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | `str` | 指标名称 |
| `value` | `float` | 原始计算值 |
| `score` | `float` | 归一化分数（0~100） |
| `weight` | `float` | 权重 |
| `detail` | `str` | 说明 |

### 4.3 四维指标计算

#### 4.3.1 流量偏差率（权重 30%）

**计算公式**：
```
deviation_rate = |actual_flow - rolling_mean| / rolling_mean
avg_deviation = mean(deviation_rate)
```
- `rolling_mean`: 前30分钟滑动窗口均值（作为"期望流量"的近似）
- 只使用 `is_valid=True` 的数据，最少需要30个有效点

**评分映射**：

| 平均偏差率 | 分数 | 含义 |
|---|---|---|
| <5% | 100 | 几乎无偏差 |
| 5%~10% | 80~100 | 轻微偏差 |
| 10%~20% | 60~80 | 中等偏差 |
| 20%~50% | 30~60 | 较大偏差 |
| >50% | 0~30 | 严重偏差 |

**与设计文档对比**：设计文档使用 `D = (Qc - Qx) / Qx`（实际流量 vs 泵效率曲线流量）。由于没有泵效率曲线数据，实现中使用滑动窗口均值代替 `Qx`。**偏差公式含义一致**（衡量实际输出与期望值的偏离程度），但参考基准不同。

#### 4.3.2 故障频率（权重 30%）

**计算公式**：
```
fault_count = count(error == 1)
fault_per_day = fault_count / (total_hours / 24)
```
- `total_hours`: 数据时间跨度（小时）

**评分映射**：

| 故障频率（次/天） | 分数 | 含义 |
|---|---|---|
| 0 | 100 | 无故障 |
| 0~1 | 80~100 | 偶发故障 |
| 1~5 | 50~80 | 较频繁故障 |
| >5 | 0~50 | 高频故障 |

**与设计文档对比**：设计文档通过PLC故障信号直接映射健康状态（二类/三类）。实现中将故障次数量化为连续评分，更细粒度。**报警触发逻辑一致**（error=1 即为故障），评分粒度不同。

#### 4.3.3 运行稳定性（权重 20%）

**计算公式**：
```
CV = std(flow) / mean(flow)
```
- 只使用 `is_valid=True` 且 `value > 0` 的数据（即运行中的数据）
- 最少需要30个有效运行点

**评分映射**：

| 变异系数(CV) | 分数 | 含义 |
|---|---|---|
| <0.05 | 100 | 极其稳定 |
| 0.05~0.10 | 80~100 | 稳定 |
| 0.10~0.30 | 50~80 | 有波动 |
| >0.30 | 0~50 | 严重波动 |

**与设计文档对比**：设计文档未明确定义"稳定性"计算方法，CV（变异系数）是工程上衡量稳定性的标准方法，属于合理实现。

#### 4.3.4 控制响应性（权重 20%）

**计算公式**：
```
consistency_rate = count(auto == remote) / total_count
```
- 将 `auto` 和 `remote` 按时间戳 inner join
- 运行时 `auto` 和 `remote` 应同时为1（同时为0表示停机，也一致）

**评分映射**：

| 状态一致率 | 分数 | 含义 |
|---|---|---|
| >95% | 100 | 高度一致 |
| 80%~95% | 70~100 | 基本一致 |
| <80% | 0~70 | 状态混乱 |

**与设计文档对比**：设计文档中"控制响应性"指自动/远控指令执行的及时性和一致性。实现中用 `auto == remote` 的匹配率来衡量，逻辑合理。

### 4.4 综合评分

```
overall_score = Σ(indicator_score × indicator_weight)
             = flow_deviation_score × 0.30
             + fault_frequency_score × 0.30
             + stability_score × 0.20
             + responsiveness_score × 0.20
```

### 4.5 健康等级

| 分数范围 | 等级 | 对应设计文档 |
|---|---|---|
| 90~100 | 优秀 | 一类 (100~80) |
| 70~90 | 良好 | 一类偏下 |
| 50~70 | 轻微异常 | 二类 (80~60) |
| 0~50 | 异常 | 三类 (60~40) |

**与设计文档对比**：设计文档三类对应 100~80 / 80~60 / 60~40。实现的分界线为 90/70/50，分界值略高于设计文档，更保守（更容易触发告警）。这是合理的工程选择——提前预警优于事后报警。

### 4.6 设计文档中未实现的部分

| 功能 | 原因 |
|---|---|
| **"短板原则"（故障钳制分数上下限）** | 当前为纯加权评分，无故障信号钳制逻辑。设计文档要求故障时限制分数范围 |
| **修复后恢复逻辑（60分起步+每小时+0.5分）** | 需要故障修复事件数据，当前数据中无此信息 |
| **设备运行时长指标** | 设计文档中 `T = (计划维护时间 - 上次维护时间) / 计划维护时间`，需维护记录数据，当前不可用 |
| **产能评估（最大有效产能/负荷率/可调余量）** | 需要泵的额定流量数据，当前不可用 |

## 5. 设备诊断模块

**文件**: `equipment_diagnosis.py`

### 5.1 输入

| 参数 | 类型 | 说明 |
|---|---|---|
| `date_range` | `tuple[str, str]` | 诊断日期范围 |
| `pump_id` | `str` | 泵编号（仅泵诊断需要） |

### 5.2 输出

**`list[EquipmentDiagnosisResult]`**，每条结果包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `category` | `str` | 诊断类别 |
| `target` | `str` | 诊断对象 |
| `anomaly_type` | `str` | 异常类型编码 |
| `alarm_level` | `int` | 报警等级 1/2/3 |
| `alarm_level_name` | `str` | 等级名称 |
| `start_time` | `datetime` | 异常开始时间 |
| `end_time` | `datetime` | 异常结束时间 |
| `duration_min` | `int` | 持续时间（分钟） |
| `detail` | `str` | 详细描述 |

结果按 `alarm_level` 升序排列（一级优先）。

### 5.3 五类诊断算法

#### 5.3.1 投加泵异常诊断 `diagnose_pump()`

**数据源**: `load_pump_data(pump_id)` → `flow`, `error`, `auto`, `remote`

**检测项一：泵故障信号 (`pump_fault`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| `error == 1` 连续出现 | 二级 | ⚠️ 设计文档将泵故障归为"三类"（影响设备使用），实现中为二级 |

**算法**：
1. 筛选 `is_valid=True` 且 `value == 1` 的故障点
2. 按连续性分组
3. 每组生成一条诊断结果

**检测项二：自动模式无流量 (`no_flow_in_auto`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| `auto == 1` 且 `flow ≤ 0`，持续≥5分钟 | 一级 | ✅ 合理，泵在自动模式运行但无输出是严重异常 |

**算法**：
1. Inner join `flow` 和 `auto` 数据
2. 筛选 `auto == 1` 且 `flow ≤ 0` 的点
3. 按连续性分组，过滤持续≥5分钟的段

**检测项三：远控/自动状态不一致 (`state_inconsistent`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| `auto != remote`，持续≥10分钟 | 三级 | ✅ 合理，状态不匹配但泵仍可能运行 |

**算法**：
1. Inner join `auto` 和 `remote` 数据
2. 筛选 `auto != remote` 的点
3. 按连续性分组，过滤持续≥10分钟的段

#### 5.3.2 投加管道异常诊断 `diagnose_pipeline()`

**数据源**: 各泵 `load_pump_data()` 的 `flow` + `load_sensor_data("投加流量计")`

**检测项一：管道堵塞 (`pipe_blockage`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 泵有输出(`flow > 0`)但管道流量计读数 < 泵流量×10%，持续≥5分钟 | 二级 | ⚠️ 设计文档用压力>0.3MPa判断堵塞，实现中用流量对比（因无压力数据） |

**算法**：
1. 加载各泵流量和对应管道流量计数据
2. Inner join 后筛选 `pipe_flow < pump_flow × 0.1` 的点
3. 按连续性分组，过滤持续≥5分钟的段

**检测项二：管道泄漏 (`pipe_leak`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 储液池液位变化率 < -0.5cm/min，持续≥10分钟 | 一级 | ⚠️ 设计文档使用 `α = (Δh×S - ∫Q) / (Δh×S)` 公式，实现中简化为液位下降速率检测 |

**算法**：
1. 加载储液池液位计数据
2. 计算分钟级液位变化 `diff(value)`
3. 筛选变化率 < -0.5 的点
4. 按连续性分组，过滤持续≥10分钟的段
5. 计算总下降量

#### 5.3.3 阀门异常诊断 `diagnose_valve()`

**数据源**: `load_sensor_data("阀门")` → 3个阀门 × 3个信号（关到位/开到位/故障汇总）

**检测项一：阀门故障 (`valve_fault`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 故障汇总信号 = 1 | 二级 | ✅ 设计文档：阀门超时为二级报警 |

**检测项二：阀门卡死 (`valve_stuck`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 关到位=0 且 开到位=0（阀门处于中间位置），持续≥3分钟 | 三级 | ⚠️ 设计文档中"未完全开启/关闭"为一级，实现中为三级 |

**检测项三：信号矛盾 (`valve_signal_conflict`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 关到位=1 且 开到位=1（信号矛盾） | 二级 | ✅ 合理，信号矛盾说明传感器或PLC异常 |

#### 5.3.4 储液池异常诊断 `diagnose_tank()`

**数据源**: `load_sensor_data("储液池液位计")` → 3个储液池液位

**检测项一：液位超高 (`level_high`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 液位 > 3.0m，持续≥5分钟 | 二级 | ✅ 合理 |

**检测项二：液位超低 (`level_low`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 液位 < 0.3m | 二级 | ✅ 设计文档：液位降至0.3m需紧急切换 |
| 液位 < 0.1m（极低） | 一级 | ✅ 合理，极低液位更严重 |

**检测项三：液位变化速率异常 (`level_fast_change`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 液位变化率 > 0.3m/min，持续≥5分钟 | 三级 | ✅ 合理 |

**阈值设置**：

| 阈值 | 值 | 说明 |
|---|---|---|
| `LEVEL_HIGH` | 3.0m | 高液位报警线 |
| `LEVEL_LOW` | 0.3m | 低液位报警线 |
| `LEVEL_CRITICAL_LOW` | 0.1m | 极低液位（一级报警） |
| 快速变化阈值 | 0.3m/min | 异常变化速率 |

#### 5.3.5 PLC通讯中断诊断 `diagnose_plc_communication()`

**数据源**: `load_csv_data()` 直接读取 ZHJY/ShuiZhi/shuizhi 三个 tagtable 的代表性点位

**检测项一：通讯完全中断 (`comm_total_loss`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 指定时间范围内完全无数据 | 一级 | ✅ 设计文档：通讯超时>3次或无心跳 → 报警 |
| 所有数据均为异常标记值 | 一级 | ✅ |

**检测项二：通讯间歇中断 (`comm_intermittent`)**

| 检测条件 | 报警等级 | 与设计文档一致性 |
|---|---|---|
| 数据覆盖率 < 90%（缺失>10%的小时） | 二级 | ⚠️ 设计文档未定义具体覆盖率阈值，10%为工程经验值 |

**算法**：
1. 加载各数据源的代表性点位数据
2. 检查数据是否完全为空或全部无效
3. 计算按小时的数据覆盖率
4. 覆盖率 < 90% 则报间歇中断

### 7.2 合理调整的部分 ⚠️

| 功能 | 设计文档 | 实现 | 调整原因 |
|---|---|---|---|
| 健康度综合评分 | `α×D + β×T` (α=0.6, β=0.4) | 四维加权 (0.30/0.30/0.20/0.20) | 无运行时长数据，增加稳定性和响应性维度 |
| 流量偏差参考值 | 泵效率曲线流量 `Qx` | 滑动窗口均值 | 无泵效率曲线数据 |
| 管道堵塞检测 | 压力 > 0.3MPa | 泵有输出但管道流量近0 | 无压力传感器数据 |
| 管道泄漏检测 | `α = (Δh×S - ∫Q) / (Δh×S)` | 液位下降速率 > 0.5cm/min | 无储液池截面积数据，简化为速率检测 |
| 健康等级分界 | 80/60/40 | 90/70/50 | 更保守的预警策略 |
| 阀门卡死等级 | 一级 | 三级 | 中间位置≠不能使用，降低等级 |
